# NavigationContainer

NavigationContainer 负责管理你的应用状态，并将你的顶级导航器链接到应用环境。

容器负责特定于平台的集成，并提供各种有用的功能:

  1. 深度链接集成与链接道具。
  2. 通知状态更改以进行屏幕跟踪 (opens new window)、状态持久化 (opens new window)等。
  3. 在 Android 上使用 React Native 中的 BackHandler (opens new window) API 处理系统后退按钮。

使用方法:

import { NavigationContainer } from "@react-navigation/native";
import { createStackNavigator } from "@react-navigation/stack";

const Stack = createStackNavigator();

export default function App() {
  return (
    <NavigationContainer>
      <Stack.Navigator>{/* ... */}</Stack.Navigator>
    </NavigationContainer>
  );
}

# Ref

还可以向容器附加 ref 以获得对各种可用的方法的访问，例如调度导航操作。

栗子：

function App() {
  const navigationRef = React.useRef(null);

  return (
    <View style={{ flex: 1 }}>
      <Button onPress={() => navigationRef.current?.navigate("Home")}>
        Go home
      </Button>
      <NavigationContainer ref={navigationRef}>{/* ... */}</NavigationContainer>
    </View>
  );
}

请记住，在某些情况下(如启用链接时)，ref 最初可能是空的。为了确保 ref 已经初始化，可以使用 onReady 回调在导航容器完成挂载时得到通知。

# ref 上的方法

ref 对象包含了所有常见的导航方法，如 navigation, goBack 等。更多细节请参见文档 CommonActions (opens new window)。

栗子：

navigationRef.current?.navigate(name, params);

所有这些方法的行为就好像它们是在当前聚焦的屏幕中被调用的一样。需要注意的是，必须渲染一个导航器来处理这些操作。

# resetRoot

resetRoot 方法可以将导航树的状态重置为指定的状态对象:

navigationRef.current?.resetRoot({
  index: 0,
  routes: [{ name: "Profile" }],
});

与 reset 方法不同的是，它作用于根导航器，而不是当前聚焦屏幕的导航器。

# getRootState

getRootState 方法返回一个导航状态对象，其中包含导航树中所有导航器的导航状态:

const state = navigationRef.current?.getRootState();

请注意，如果当前没有显示导航器，返回的 state 对象将是 undefined。

# getCurrentRoute

getCurrentRoute 方法返回整个导航树中当前聚焦屏幕的路由对象:

const route = navigationRef.current?.getCurrentRoute();

注意，如果当前没有渲染导航器，返回的 route 对象将是 undefined。

# getCurrentOptions

getCurrentOptions 方法返回整个导航树中当前聚焦屏幕的选项:

const options = navigationRef.current?.getCurrentOptions();

请注意，如果当前没有显示导航器，返回的 options 对象将是 undefined。

# addListener

addListener 方法让你监听以下事件:

# state

当导航树中的导航状态发生变化时，该事件会被触发:

const unsubscribe = navigationRef.current?.addListener("state", (e) => {
  // You can get the raw navigation state (partial state object of the root navigator)
  console.log(e.data.state);

  // Or get the full state object with `getRootState()`
  console.log(navigationRef.current.getRootState());
});

这类似于 onStateChange 方法。唯一的区别是 e.data.state 对象可能包含部分状态对象，而 onStateChange 中的 state 参数总是包含完整的状态对象。

# options

当导航树中当前焦点屏幕的选项发生变化时，该事件就会被触发:

const unsubscribe = navigationRef.current?.addListener("options", (e) => {
  // You can get the new options for the currently focused screen
  console.log(e.data.options);
});

# Props

# initialState

接受导航器初始状态。这对于深度链接、状态持久化等情况非常有用。

例子:

<NavigationContainer
  onStateChange={(state) => console.log("New state is", state)}
  initialState={initialState}
>
  {/* ... */}
</NavigationContainer>

提供一个自定义初始状态对象将覆盖通过链接配置或从浏览器的 URL 获得的初始状态对象。如果你提供一个初始状态对象，确保你没有在网络上传递它，没有深层链接需要处理。

例子:

const initialUrl = await Linking.getInitialURL();

if (Platform.OS !== "web" && initialUrl == null) {
  // Only restore state if there's no deep link and we're not on web
}

有关如何持久化和恢复状态的更多细节，请参见状态 持久化指南。

# onStateChange

注意:考虑导航器的状态对象是内部的，并且在小版本中会发生更改。除非确实需要，否则避免使用来自导航状态对象的属性 (除了 index 和 routes)。如果不依赖状态对象的结构就无法实现某些功能，请提出一个问题。

每次导航状态改变时调用的函数。它接收新的导航状态作为参数。

您可以使用它来跟踪焦点屏幕，持久化导航状态等。

# onReady

在导航容器及其所有子容器第一次完成挂载后调用的函数。你可以用它来:

  • 确保 ref 是可用的。参见文档中关于初始化引用的更多细节。
  • 隐藏您的原生启动画面

# linking

用于深度链接的链接集成配置，浏览器中的 URL 支持等。

栗子：

import { NavigationContainer } from "@react-navigation/native";

function App() {
  const linking = {
    prefixes: ["https://mychat.com", "mychat://"],
    config: {
      screens: {
        Home: "feed/:sort",
      },
    },
  };

  return (
    <NavigationContainer linking={linking} fallback={<Text>Loading...</Text>}>
      {/* content */}
    </NavigationContainer>
  );
}

关于如何 配置深度链接 和 URL 集成的更多细节，请参见配置链接指南。

# Options

# linking.prefixes

要处理的 URL 前缀。您可以提供多个前缀来支持自定义方案和通用链接。

只处理匹配这些前缀的 url。在解析之前，前缀将从 URL 中剥离。

栗子：

<NavigationContainer
  linking={{
    prefixes: ["https://mychat.com", "mychat://"],
    config: {
      screens: {
        Chat: "feed/:sort",
      },
    },
  }}
>
  {/* content */}
</NavigationContainer>

这只支持 iOS 和 Android。

# linking.config

配置以调整如何解析路径。config 对象应该代表应用程序中导航器的结构。

例如，如果我们在主屏幕中有 Catalog 屏幕，并希望它处理 item/:id 模式:

{
  screens: {
    Home: {
      screens: {
        Catalog: {
          path: 'item/:id',
          parse: {
            id: Number,
          },
        },
      },
    },
  }
}

解析的选项可以是对象或字符串:

{
  screens: {
    Catalog: 'item/:id',
  }
}

当指定一个字符串时，它相当于提供 path 选项。

路径选项是与路径匹配的模式。任何以 : 开头的段都被识别为具有相同名称的参数。例如，item/42 将被解析为{name: 'item'， params: {id: '42'}}。

initialRouteName 选项确保在那里传递的路由名会出现在导航器的状态中，例如 config:

{
  screens: {
    Home: {
      initialRouteName: 'Feed',
      screens: {
        Catalog: {
          path: 'item/:id',
          parse: {
            id: Number,
          },
        },
        Feed: 'feed',
      },
    },
  }
}

URL: /item/42，状态是这样的:

{
  routes: [
    {
      name: 'Home',
      state: {
        index: 1,
        routes: [
          {
            name: 'Feed'
          },
          {
            name: 'Catalog',
            params: { id: 42 },
          },
        ],
      },
    },
  ],
}

parse 选项控制如何解析参数。在这里，你可以提供参数的名称作为键来解析，以及一个函数，它接受参数的字符串值并返回一个解析后的值:

{
  screens: {
    Catalog: {
      path: 'item/:id',
      parse: {
        id: id => parseInt(id, 10),
      },
    },
  }
}

如果没有提供用于解析参数的自定义函数，它将被解析为字符串。

# linking.enabled

可选布尔值，用于启用或禁用链接集成。如果指定了 linking prop，则默认为 true。

# linking.getInitialURL

默认情况下，linking 集成了 React Native 的 Linking API，并使用了 Linking.getInitialURL() 来提供对深度链接的内置支持。但是，你也可能想要处理来自其他来源的链接，比如 Branch (opens new window)，或者使用 Firebase (opens new window) 等推送通知。

您可以提供一个自定义的 getInitialURL 函数，在那里您可以返回我们应该用作初始 URL 的链接。如果有一个 URL 要处理，getInitialURL 函数应该返回一个string，否则 undefined。

例如，你可以像下面这样处理深度链接和 Firebase 通知 (opens new window):

<NavigationContainer
  linking={{
    prefixes: ["https://mychat.com", "mychat://"],
    config: {
      screens: {
        Chat: "feed/:sort",
      },
    },
    async getInitialURL() {
      // Check if app was opened from a deep link
      const url = await Linking.getInitialURL();

      if (url != null) {
        return url;
      }

      // Check if there is an initial firebase notification
      const message = await messaging().getInitialNotification();

      // Get the `url` property from the notification which corresponds to a screen
      // This property needs to be set on the notification payload when sending it
      return message?.notification.url;
    },
  }}
>
  {/* content */}
</NavigationContainer>

此选项在 Web 上不可用。

# linking.subscribe

与 getInitialURL 类似，您可以提供一个自定义 subscribe 来处理任何传入链接，而不是默认的深度链接处理。subscribe 将接收一个侦听器作为参数，当有新的 URL 要处理时，你可以用 URL 字符串调用它。它应该返回一个清理函数，您可以在其中取消订阅已设置的任何事件侦听器。

例如，你可以像下面这样处理深度链接和 Firebase 通知 (opens new window):

<NavigationContainer
  linking={{
    prefixes: ["https://mychat.com", "mychat://"],
    config: {
      screens: {
        Chat: "feed/:sort",
      },
    },
    subscribe(listener) {
      const onReceiveURL = ({ url }: { url: string }) => listener(url);

      // Listen to incoming links from deep linking
      Linking.addEventListener("url", onReceiveURL);

      // Listen to firebase push notifications
      const unsubscribeNotification = messaging().onNotificationOpenedApp(
        (message) => {
          const url = message.notification.url;

          if (url) {
            // Any custom logic to check whether the URL needs to be handled
            //...

            // Call the listener to let React Navigation handle the URL
            listener(url);
          }
        }
      );

      return () => {
        // Clean up the event listeners
        Linking.removeEventListener("url", onReceiveURL);
        unsubscribeNotification();
      };
    },
  }}
>
  {/* content */}
</NavigationContainer>

此选项在 Web 上不可用。

例子:

<NavigationContainer
  linking={{
    prefixes: ["https://mychat.com", "mychat://"],
    config: {
      screens: {
        Chat: "feed/:sort",
      },
    },
    getStateFromPath(path, config) {
      // Return a state object here
      // You can also reuse the default logic by importing `getStateFromPath` from `@react-navigation/native`
    },
  }}
>
  {/* content */}
</NavigationContainer>
# linking.getPathFromState

您可以选择通过提供自己的实现来覆盖 React Navigation 序列化状态对象以进行链接的方式。 如果您指定了 getStateFromPath，这对于适当的 Web 支持是必要的。

例子：

<NavigationContainer
  linking={{
    prefixes: ["https://mychat.com", "mychat://"],
    config: {
      screens: {
        Chat: "feed/:sort",
      },
    },
    getPathFromState(state, config) {
      // Return a path string here
      // You can also reuse the default logic by importing `getPathFromState` from `@react-navigation/native`
    },
  }}
>
  {/* content */}
</NavigationContainer>

# fallback

在解析深层链接时，React Element 用作备用。 默认为空。

如果您有原生启动画面，请使用 onReady 代替 fallback prop。

# documentTitle

默认情况下，React Navigation 自动更新 Web 上的文档标题以匹配焦点屏幕的标题选项。 您可以使用此道具禁用或自定义它。 它接受带有以下选项的配置对象：

# documentTitle.enabled

是否应启用文档标题处理。 默认为 true

# documentTitle.formatter

如果要自定义标题文本，请使用自定义格式程序。 默认为：

(options, route) => options?.title ?? route?.name;

例子

import { NavigationContainer } from "@react-navigation/native";

function App() {
  return (
    <NavigationContainer
      documentTitle={{
        formatter: (options, route) =>
          `${options?.title ?? route?.name} - My Cool App`,
      }}
    >
      {/* content */}
    </NavigationContainer>
  );
}

# theme

用于导航组件（如标题，标签栏等）的自定义主题。有关更多详细信息和使用指南，请参阅主题指南。
